### Q: Tailscale key expiry 怎么关

#### 结论先行
- 这是在 Tailscale admin console 里对“这台 Arch 机器”关闭 machine key expiry，不是在 Arch 本机用 `systemctl` 改。
- 目的：这台机器像 always-on server，避免过期后手机/Termius 远程连不上。
- 代价：设备丢失或 key 泄漏时风险更高；丢设备要立刻在 admin console 移除机器。

#### 入口
../arch/readme.md:211:1 — `Tailscale + 手机 Termius SSH` — 本仓库对远程回连方案的主说明
../arch/readme.md:235:1 — `Arch 端第一次配置` — `tailscale up --operator="$USER" --qr` 后机器会出现在 Machines
../arch/readme.md:254:1 — `Disable key expiry` — README 说明必须在 admin console 里对这台机器关闭

#### 主要调用顺序
../arch/readme.md:237:1 — Arch 上先 `sudo tailscale up --operator="$USER" --qr` 登录 tailnet
../arch/readme.md:246:1 — 用 `tailscale status` / `tailscale ip -4` 确认机器已加入 tailnet
../arch/readme.md:254:1 — 去 Tailscale admin console：Machines -> 这台机器 -> Disable key expiry

#### 相关测试

#### 放大 / 忽略
- 放大：Tailscale admin console 的 Machines 页面、这台 Arch 机器的 hostname / `100.x.y.z` IP。
- 暂时忽略：Tailscale SSH、API key expiry、auth key expiry；README 推荐的是 OpenSSH over Tailscale。

### Q: README 整体是不是统一 OpenSSH over Tailscale 安全方案

#### 结论先行
- 是，`arch/readme.md` 当前主线就是 OpenSSH over Tailscale：Tailscale 只做私有网络，登录认证交给 OpenSSH。
- 对你的目标来说这是专业的 baseline：远程可恢复、入口面小、key-only、LAN 不暴露、tmux 任务不断。
- 不是“绝对最安全”的学术方案；它是安全、可用、低延迟之间更合理的工程取舍。
